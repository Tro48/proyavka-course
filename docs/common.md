# Общая база для трёх проектов

Документ описывает решения, одинаковые для всех трёх лендингов. В планах проектов
(`01`, `02`, `03`) описано только то, что специфично для конкретной работы.

## Роли проектов в портфолио

Три работы не должны выглядеть как один шаблон, перекрашенный три раза. У каждой —
своя техническая «изюминка» и свой визуальный язык. Порядок реализации — по
возрастанию сложности, каждый следующий проект переиспользует навык предыдущего.

| # | Проект | Кодовое имя | Главный технический навык | Оценка |
|---|--------|-------------|---------------------------|--------|
| 1 | Барбершоп | Кузня | Интеграция с внешним каналом (Telegram Bot API), серверная валидация, аналитика целей | 16–20 ч |
| 2 | Курс фотографии | Проявка | Состояние (Zustand), время как источник истины, Supabase, условная логика потоков | 24–28 ч |
| 3 | Агентство | Стапель | Multi-step форма с персистом, TanStack Query, фильтрация через URL | 36–44 ч |

Суммарно ≈ 80–90 часов чистой разработки. При 3 часах в день — примерно месяц.

## Организация репозиториев

Три отдельных репозитория, а не монорепо. Причины:

- Стили и компоненты намеренно разные, общий UI-пакет переиспользовать почти нечего.
- Каждый репозиторий — самостоятельная ссылка в резюме с чистой историей коммитов.
- Деплой на Vercel из отдельного репо настраивается за минуты, без фильтров путей.

Общим остаётся **стартовый шаблон** — приватный репозиторий `landing-starter` с
настроенными TypeScript, ESLint, Prettier, Vitest, Playwright и GitHub Actions.
Создаётся один раз перед первым проектом, дальше используется как GitHub template.

## Стек

Версии не фиксирую в этом документе намеренно: он пишется заранее, а ставить нужно
актуальное на день старта. Правило — на старте каждого проекта фиксируем версии в
lock-файле и больше не трогаем мажоры до конца работы.

```bash
pnpm create next-app@latest kuznya --typescript --tailwind --app --eslint --src-dir
```

- **Next.js (App Router)** + **TypeScript** в режиме `strict`.
- **Tailwind CSS** — токены темы через CSS-переменные в `@theme`.
- **Motion** (`motion/react`) — это бывший Framer Motion, переименованный после
  отделения от Framer. Ставится как `pnpm add motion`, импорт `import { motion } from "motion/react"`.
- **React Hook Form** + **Zod** + `@hookform/resolvers` — формы и валидация.
- **Zustand** — состояние там, где оно действительно есть (проекты 2 и 3).
- **TanStack Query** — только в проекте 3, где данные реально приходят из Supabase.
- **Supabase** — хранение лидов и брифов (проекты 2 и 3).
- **Vercel** — деплой, preview-окружения на каждый PR.

Библиотеку компонентов (shadcn/ui и подобные) не подключаем. Три лендинга — это
ровно тот случай, когда своя пара примитивов (`Button`, `Input`, `Accordion`)
пишется быстрее, чем настраивается чужая тема, и лучше показывает владение вёрсткой.
Исключение — `radix-ui` примитивы для аккордеона и диалога, если нужна доступность
из коробки: клавиатурная навигация и ARIA там сделаны правильно, и переписывать их
руками в портфолио-проекте смысла нет.

## Структура проекта

Одинаковая во всех трёх репозиториях — это снижает переключение контекста.

```
src/
  app/
    layout.tsx           # шрифты, метаданные, аналитика
    page.tsx             # сборка секций
    globals.css          # токены темы, base-слой
    api/
      lead/route.ts      # приём формы (Route Handler, server-only)
    opengraph-image.tsx  # генерация OG-картинки
    sitemap.ts
    robots.ts
  components/
    sections/            # Hero, Services, Reviews… — по одной на секцию лендинга
    ui/                  # Button, Input, Accordion — примитивы без бизнес-логики
    layout/              # Header, Footer, Container
  lib/
    schemas/             # Zod-схемы, общие для клиента и сервера
    analytics.ts         # обёртка над Яндекс.Метрикой
    env.ts               # валидация переменных окружения через Zod
  content/               # тексты, услуги, отзывы, FAQ — как типизированные данные
```

Ключевое правило: **контент лежит в `content/` как типизированные объекты**, а не
хардкодится в JSX. Секция получает данные пропсами. Это даёт две вещи бесплатно:
контент можно позже вынести в CMS без переписывания вёрстки, и на собеседовании
видно, что автор отделяет данные от представления.

## Валидация: одна схема на клиент и сервер

Схемы Zod живут в `lib/schemas/` и импортируются и формой, и Route Handler'ом.
Серверная валидация **обязательна и не опциональна**: клиентская проверка — это
UX, а не защита. Запрос в `/api/lead` может прийти откуда угодно.

```ts
// lib/schemas/lead.ts
import { z } from "zod";

export const leadSchema = z.object({
  name: z.string().trim().min(2, "Слишком короткое имя").max(60),
  phone: z.string().regex(/^\+7\d{10}$/, "Формат: +7XXXXXXXXXX"),
  comment: z.string().trim().max(500).optional(),
  consent: z.literal(true, { message: "Нужно согласие на обработку данных" }),
  // honeypot: настоящий пользователь это поле не видит и не заполняет
  website: z.string().max(0).optional(),
});

export type LeadInput = z.infer<typeof leadSchema>;
```

Телефон храним и валидируем в нормализованном виде `+7XXXXXXXXXX`, а маску ввода
показываем пользователю. Нормализация — в одном месте, перед валидацией.

## Защита форм от спама

Форма с отправкой в Telegram или базу — это открытая дверь. Три уровня, в порядке
соотношения «польза / затраты»:

1. **Honeypot** — скрытое поле `website`. Боты его заполняют, люди нет. Прячем
   через CSS-класс с `position: absolute; left: -9999px` и `aria-hidden`, а не
   `display: none` (часть ботов пропускает такие поля). Стоит 10 минут.
2. **Time-trap** — форма отрисована меньше 3 секунд назад, значит это бот. В форму
   кладём `renderedAt` (timestamp), на сервере сравниваем с `Date.now()`.
3. **Rate limit по IP** — `@upstash/ratelimit` + Upstash Redis, бесплатного тарифа
   хватает с запасом. 5 запросов в час на IP. Ставим на все три проекта.

Капчу не добавляем: она портит конверсию, а для портфолио-проекта с трафиком в
десятки визитов трёх уровней выше более чем достаточно.

## Секреты и переменные окружения

Токен Telegram-бота и `SUPABASE_SERVICE_ROLE_KEY` **не должны попадать в клиент**.
Правила:

- Всё, что не начинается с `NEXT_PUBLIC_`, доступно только на сервере.
- Файл `lib/env.ts` валидирует переменные Zod-схемой при старте — приложение падает
  на билде, а не в рантайме на проде при отсутствующем токене.
- В файлы, читающие серверные секреты, добавляем `import "server-only"` — тогда
  случайный импорт из клиентского компонента сломает сборку, а не утечёт в бандл.
- В репозитории лежит `.env.example` с именами переменных без значений.

## Supabase: доступ к данным

Insert лидов делаем **только через Route Handler с service role ключом**, не с
клиента. Причина: с анонимным ключом и insert-политикой любой посетитель может
писать в таблицу напрямую из консоли браузера в цикле, а rate limit на своём
эндпоинте мы контролируем.

RLS включаем на всех таблицах. Для таблицы лидов: политик для `anon` нет вообще —
ни select, ни insert. Для таблицы кейсов (проект 3, публичные данные): политика
`select` для `anon` при `published = true`.

## Аналитика

Яндекс.Метрика через `next/script` со стратегией `afterInteractive`. Обёртка:

```ts
// lib/analytics.ts
declare global { interface Window { ym?: (id: number, action: string, goal: string) => void } }

const COUNTER_ID = Number(process.env.NEXT_PUBLIC_YM_ID);

export function reachGoal(goal: string) {
  if (typeof window === "undefined" || !window.ym || !COUNTER_ID) return;
  window.ym(COUNTER_ID, "reachGoal", goal);
}
```

Цель отправляем **после успешного ответа сервера**, а не по клику на кнопку —
иначе в статистику попадут неудачные отправки. Минимальный набор целей одинаков:
`form_submit_success`, `phone_click`, `messenger_click`.

## Доступность

Не «сделаем потом» — потом не делают. Обязательный минимум:

- Контраст текста ≥ 4.5:1, крупного текста и элементов управления ≥ 3:1. Палитры
  в планах проектов уже проверены на это.
- Видимый `:focus-visible` на всех интерактивных элементах. Не `outline: none`.
- Все анимации уважают `prefers-reduced-motion: reduce` — не отключаются целиком,
  а заменяются на мгновенную смену состояния (иначе сломается логика появления).
- Ошибки формы связаны с полем через `aria-describedby`, поле помечено `aria-invalid`.
- Порядок заголовков без дыр: один `h1` на страницу, дальше `h2` на каждую секцию.

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

## Производительность

Цели Lighthouse на мобильном (throttled, продовый билд, не dev-сервер):
Performance ≥ 90, Accessibility = 100, Best Practices ≥ 95, SEO = 100.

- Шрифты через `next/font/google` с `display: "swap"` и явным `subsets: ["cyrillic"]`.
  Без этого браузер тянет латиницу и кириллицу отдельными файлами.
- Все изображения — `next/image` с явными `width`/`height`, `sizes` и `priority`
  только на LCP-картинке героя.
- Секции ниже первого экрана анимируются по `whileInView`, а не грузят JS сразу.
- Motion импортируем гранулярно; тяжёлые компоненты (карта, галерея) — `dynamic()`
  с `ssr: false` там, где они не нужны для SEO.

**Яндекс.Карты — главный риск для метрики Performance.** Iframe карты весит больше,
чем весь остальной лендинг. Решение: рендерим статичную картинку-заглушку с кнопкой
«Показать карту», iframe монтируем по клику. Заодно это честно по отношению к
пользователю: карта грузится, когда она нужна.

## SEO и разметка

- `metadata` в `layout.tsx` и `page.tsx`, включая `openGraph` и `twitter`.
- OG-картинка генерируется через `opengraph-image.tsx` (Next.js ImageResponse), а не
  рисуется руками в Figma — это дешевле и заодно показывает знание API.
- JSON-LD в `<script type="application/ld+json">`: `LocalBusiness` для барбершопа,
  `Course` для инфопродукта, `ProfessionalService` для агентства.
- `sitemap.ts` и `robots.ts` — файловые конвенции App Router.

## Юридический минимум для форм в РФ

Форма собирает персональные данные, значит по 152-ФЗ нужны:

- Чекбокс согласия на обработку персональных данных — **не предустановленный**,
  с ссылкой на политику. В Zod-схеме это `z.literal(true)`, форма не отправится без него.
- Страница `/privacy` с политикой конфиденциальности.

Для портфолио это не формальность, а деталь, которую замечает заказчик: она
показывает, что разработчик знает, что бывает после сдачи лендинга.

## Качество кода

- `pnpm lint`, `pnpm typecheck`, `pnpm build` — блокирующие шаги в GitHub Actions
  на каждый PR. Красный CI не мержится.
- **Vitest** — юнит-тесты на Zod-схемы (граничные значения телефона, email,
  honeypot) и чистые функции (расчёт таймера, нормализация телефона). Это те места,
  где ошибка не видна глазами.
- **Playwright** — один smoke-сценарий на проект: открыть страницу, заполнить форму,
  замокать сетевой ответ, увидеть экран успеха. Не пишем E2E на всё подряд.
- Коммиты в стиле Conventional Commits. История коммитов — часть портфолио.

## Порядок работы над каждым проектом

Одинаковый во всех трёх, отклонение только в объёме шага 4:

1. Контент. Пишем все тексты в `content/` до вёрстки. Вёрстка под реальный текст —
   это другая вёрстка, чем под `lorem ipsum`.
2. Токены темы и примитивы `ui/`.
3. Секции сверху вниз, каждая — отдельный коммит.
4. Логика: интеграции, состояние, данные.
5. Доступность и производительность: прогон Lighthouse, клавиатурная навигация,
   проверка контраста.
6. Метаданные, OG, JSON-LD, sitemap.
7. Деплой, домен, проверка целей Метрики на проде.
8. README с обоснованием решений.

Не начинать следующий проект, пока предыдущий не задеплоен и не имеет README.
Три задеплоенных лендинга сильнее пяти недоделанных.
