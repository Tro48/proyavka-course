# Проект №2 — «Проявка», курс плёночной фотографии

> Читать вместе с [`common.md`](./common.md).

## Анализ

### Выбор темы

Нишевый офлайн-навык вместо курса по вёрстке. Взял **плёночную фотографию: съёмка,
ручная проявка, печать**. Причины выбора именно её:

- Тема даёт готовый визуальный язык: тёмная комната, красный фонарь, контрольные
  листы, зерно. Дизайн не придётся выдумывать из ничего.
- Курс офлайновый по своей природе (нужна лаборатория), значит поток с ограниченным
  числом мест — не маркетинговая уловка, а физическое ограничение. Таймер и «осталось
  4 места» перестают быть враньём, и лендинг честен.
- Показывает, что вы умеете работать с чужой предметной областью. Фронтендер, сделавший
  лендинг курса по React, не доказал ничего: он писал про себя.

Название потока и продукта — **«Проявка»**. Слово означает и физический процесс, и то,
что происходит с человеком на курсе. Второго уровня смысла достаточно, объяснять его
на лендинге не нужно.

### Что на самом деле продаёт лендинг

Не навык. **Результат, который можно показать другим.** Человек, идущий на курс
фотографии, хочет через два месяца иметь десять снимков, которые не стыдно повесить
на стену. Значит:

- Герой обещает не «научитесь снимать на плёнку», а **«через 8 недель — своя
  напечатанная серия»**. Осязаемо и проверяемо.
- Программа модулей описывает не темы, а **что будет на руках после каждого модуля**.
- Главное социальное доказательство — не отзыв текстом, а **работы выпускников**.

### Что меняю в ТЗ

| Было | Стало | Причина |
|------|-------|---------|
| Таймер до старта потока | Таймер + счётчик оставшихся мест | Дедлайн по времени слабее дедлайна по местам, когда мест физически мало. Вместе работают сильнее. |
| Отзывы не предусмотрены | Блок «Работы выпускников» с подписями | Для курса по визуальному навыку работы учеников убедительнее текста. |
| Тарифы: базовый / про / с наставником | Те же три, но «Про» выделен и объяснён | Добавляю строку «что выбирают 7 из 10» — это факт о продукте, а не давление. |
| Форма: email + имя | То же + телефон опционально | Для офлайн-курса нужен способ связаться быстро, но обязательным делать нельзя: телефон снижает конверсию сильнее, чем помогает. |

### Техническая ценность для портфолио

Проект про **состояние и время**. Три вещи, которые здесь делаются правильно и
почти везде делаются неправильно:

1. **Таймер не верит клиенту.** `Date.now()` в браузере — это системное время
   пользователя, которое он может перевести. Дедлайн приходит с сервера.
2. **Таймер не ломает гидратацию.** Сервер и клиент отрендерят разное время, React
   выдаст hydration mismatch. Решение ниже, в разделе про таймер.
3. **Условная логика потоков.** Поток стартовал — таймер переключается на следующий,
   меняются тексты кнопок и заголовка. Это состояние, производное от времени, и
   считать его надо в одном месте, а не в пяти компонентах.

Zustand здесь оправдан: выбранный тариф нужен одновременно карточкам тарифов,
липкой панели с ценой и форме брони. Прокидывать это пропсами через пять уровней —
именно тот случай, ради которого стор и придуман.

### Риски

- **Zustand ради Zustand.** Соблазн положить в стор всё, включая тики таймера. Тик
  раз в секунду через глобальный стор перерисует всех подписчиков. Тикающее значение
  живёт в локальном `useState` внутри `<Countdown/>`, в сторе — только дедлайн и статус.
- **Часовые пояса.** Даты потоков задаём в UTC, показываем в московском времени с
  явной подписью «(мск)». Пользователь из Владивостока не должен гадать.
- **Поток в прошлом.** Через полгода после деплоя все потоки в конфиге истекут и
  лендинг покажет пустоту. Нужен fallback-режим, см. ниже. Для портфолио это
  критично: работодатель откроет ссылку через год.

## Дизайн

### Идея

**Тёмная комната и контрольный лист.** Страница начинается в темноте — герой почти
чёрный, с красным светом фонаря. По мере скролла страница «проявляется»: секции
становятся светлее, и нижняя часть (тарифы, форма) — уже бумага, тёплая и светлая.
Метафора процесса, положенная на скролл.

Это единственный трюк на лендинге, и он бесплатный: два цвета фона и `background`
на секциях, никакого JS.

### Палитра

```css
@theme {
  /* тёмная часть — проявочная */
  --color-darkroom: #14100F; /* фон героя */
  --color-safelight: #8C2A22; /* красный фонарь, только свечение и акценты */
  --color-glow:      #C4402F; /* активное состояние на тёмном */

  /* светлая часть — бумага */
  --color-paper:  #FAF8F5; /* фон нижних секций */
  --color-ink:    #1C1A17; /* текст на бумаге */
  --color-graphite: #6B6459; /* вторичный текст */
  --color-terracotta: #A83E28; /* акцент на светлом: кнопки, ссылки */

  --color-line: #E2DCD3; /* границы на светлом */
  --color-danger: #B3261E;
}
```

Контраст: `ink` на `paper` — 14.9:1. `graphite` на `paper` — 5.4:1. `terracotta` на
`paper` — 6.1:1, и белый текст на `terracotta` — 5.9:1, кнопка проходит AA.
`glow` на `darkroom` — 5.2:1. `safelight` на `darkroom` — 3.4:1, **для текста не
использовать**, только для свечений, рамок и декоративных элементов.

Последнее — типичная ошибка: красивый тёмно-красный ставят в подпись под заголовком
на чёрном, и её невозможно прочитать. `safelight` в этом проекте — источник света,
а не краска для букв.

### Типографика

- Заголовки: **Cormorant Garamond** 600, кириллица есть. Антиква с высоким контрастом
  штриха — «фотографическая», отсылает к книгам по искусству.
- Текст: **Inter** 400/500. Гротеск против антиквы — нужный контраст пар.
- Цифры таймера и номера модулей: **JetBrains Mono** 500, `tabular-nums`.
  Моноширинный шрифт в таймере обязателен: иначе цифры прыгают при каждой смене
  секунды, потому что «1» уже «8».

```css
--text-hero: clamp(2.5rem, 6.5vw, 5rem);  /* Cormorant, line-height: 1.05 */
--text-h2:   clamp(1.75rem, 3.5vw, 2.75rem);
--text-body: 1.0625rem;                    /* line-height: 1.7 — длинные тексты программы */
--text-mono: 0.875rem;
```

### Детали

- **Зерно.** Полупрозрачный SVG-шум (`feTurbulence`) поверх героя, `opacity: 0.04`,
  `pointer-events: none`. Один инлайновый SVG, не картинка. Даёт плёночную фактуру.
- **Контрольный лист.** Секция «Работы выпускников» свёрстана как contact sheet:
  плотная сетка кадров с тонкими белыми полями и подписями моноширинным шрифтом.
- **Скругления** `2px`. Фотобумагу не скругляют.
- **Таймер** — четыре моноширинных блока с подписями «дней / часов / минут / секунд»,
  цифры цветом `glow` на `darkroom`. Секунды не выделяем — они отвлекают от дней.

### Анимации

- Проявление изображений при появлении в вьюпорте: `filter: blur(12px) → blur(0)` +
  `opacity 0 → 1`, `duration: 0.8`. Ровно та анимация, которая описывает предмет курса.
  Не применять к тексту — только к фото.
- Аккордеон программы: анимация высоты через `motion` с `layout`, `duration: 0.25`.
- Смена тарифа: подсветка границы, без масштабирования карточки.
- При `prefers-reduced-motion` blur-проявление отключается — фото просто на месте.

## Модель потоков

Сердце проекта. Один типизированный конфиг, из него выводится всё остальное.

```ts
// content/cohorts.ts
export type Cohort = {
  id: string;
  number: number;
  startsAt: string;   // ISO 8601, UTC: "2026-09-15T15:00:00Z"
  endsAt: string;
  seatsTotal: number;
  seatsLeft: number;
  enrollmentClosesAt: string; // приём заявок закрывается раньше старта
};

export const cohorts: Cohort[] = [ /* 3–4 потока вперёд */ ];
```

Производное состояние считается **одной чистой функцией**, которую легко покрыть
тестами и невозможно рассинхронизировать между компонентами:

```ts
// lib/cohort.ts
export type CohortState =
  | { kind: "enrolling"; cohort: Cohort; deadline: string }   // идёт набор
  | { kind: "lastCall"; cohort: Cohort; deadline: string }    // < 48 ч или ≤ 3 мест
  | { kind: "closed"; current: Cohort; next: Cohort; deadline: string } // набор закрыт, есть следующий
  | { kind: "waitlist" };                                      // потоков в конфиге не осталось

export function resolveCohortState(now: Date, list: Cohort[]): CohortState { /* … */ }
```

Что меняется в каждом состоянии:

| Состояние | Заголовок героя | Таймер | Кнопка |
|-----------|-----------------|--------|--------|
| `enrolling` | «Старт 15 сентября» | до закрытия набора | «Забронировать место» |
| `lastCall` | «Осталось 3 места» | до закрытия набора, красный | «Успеть записаться» |
| `closed` | «Набор на поток 4 закрыт» | до старта следующего | «Записаться на поток 5» |
| `waitlist` | «Следующий поток — весной» | нет | «В лист ожидания» |

`waitlist` — тот самый fallback, который спасает лендинг через год после деплоя.
Без него страница покажет отрицательный таймер, и это первое, что заметит
работодатель, открывший ссылку из резюме.

## Таймер: три ловушки

**1. Hydration mismatch.** Сервер рендерит `03:12:44`, клиент через 200 мс — `03:12:43`.
React ругается. Правильно: на сервере рендерим статичную разметку-плейсхолдер,
живое значение считаем после монтирования.

```tsx
"use client";
export function Countdown({ deadline }: { deadline: string }) {
  const [left, setLeft] = useState<Duration | null>(null); // null на сервере и первом рендере

  useEffect(() => {
    const target = new Date(deadline).getTime();
    const tick = () => setLeft(toDuration(target - Date.now()));
    tick();
    const id = setInterval(tick, 1000);
    return () => clearInterval(id);
  }, [deadline]);

  // одинаковая разметка на сервере и клиенте, разное содержимое после mount
  return <TimerLayout value={left} />; // при null — прочерки той же ширины
}
```

Прочерки той же ширины, что и цифры (моноширинный шрифт) — таймер не «дёргает»
макет при появлении. Это и есть отсутствие layout shift, за который Lighthouse
снимает баллы.

**2. Клиентское время.** Пользователь с переведёнными часами увидит чужой таймер.
Дедлайн приходит с сервера как ISO-строка (RSC отдаёт его пропсом), но `Date.now()`
всё равно клиентский. Компромисс, разумный для лендинга: один раз при монтировании
запрашиваем `/api/time`, считаем `offset = serverNow - clientNow`, дальше все тики
корректируем на `offset`. Стоит один лёгкий запрос и закрывает 99% случаев.

**3. Вкладка в фоне.** Браузер троттлит `setInterval` до раза в минуту. При возврате
на вкладку таймер отстаёт. Лечится тем, что мы каждый тик считаем **разницу с
дедлайном**, а не декрементируем счётчик. Никогда не писать `setLeft(l => l - 1)`.

## Zustand: что кладём в стор

Ровно два поля. Всё остальное — производное или локальное.

```ts
// lib/store/course.ts
type CourseStore = {
  selectedPlan: PlanId;        // нужен карточкам, липкой панели и форме
  setPlan: (id: PlanId) => void;
  isBookingOpen: boolean;      // модалка формы
  openBooking: (plan?: PlanId) => void;
  closeBooking: () => void;
};
```

Чего в сторе **нет** и почему:

- Тиков таймера — перерисовали бы всё дерево раз в секунду.
- `CohortState` — производное от времени и конфига, считается через `useMemo` или
  на сервере. Хранить производное состояние — способ получить рассинхрон.
- Значений полей формы — этим занимается React Hook Form, у него для этого есть всё.

Подписка на срезы, не на весь стор: `useCourseStore(s => s.selectedPlan)`. Иначе
любое изменение стора перерисует каждый компонент, который его тронул.

## Структура страницы

1. **Hero** — тёмный. Обещание результата, дата старта, таймер, кнопка. Зерно поверх.
2. **Что будет на руках** — 3 карточки: проявленные плёнки, контрольные листы,
   напечатанная серия из 10 работ. Осязаемый результат вместо «навыков».
3. **Программа** — 8 модулей аккордеоном. Заголовок модуля = результат недели.
4. **Кому подходит** — 3 сегмента: снимает на цифру и скучно; купил плёночник и он
   лежит; хочет печатать руками. Формулировки от лица человека, не «новичок / средний».
5. **Работы выпускников** — контрольный лист, начало светлой части страницы.
6. **Автор** — фото, экспертиза, 2–3 факта с числами (лет снимает, выставок, учеников).
7. **Тарифы** — 3 карточки. «Про» выделен, у каждой кнопка открывает форму с
   предвыбранным тарифом.
8. **FAQ** — аккордеон, 6 вопросов. Обязательно: «а если пропущу занятие», «нужна ли
   своя камера», «вернёте ли деньги».
9. **Форма брони** — модалка или секция, зависит от точки входа.
10. **Footer** — контакты, `/privacy`, `/offer`, пометка о демо.

**Липкая нижняя панель на мобильном**: цена выбранного тарифа + кнопка. Появляется
после скролла за тарифы. Это то место, где `selectedPlan` из Zustand окупается.

## Supabase

```sql
create table public.leads (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  name text not null check (char_length(name) between 2 and 60),
  email text not null check (position('@' in email) > 1),
  phone text,
  plan text not null check (plan in ('base', 'pro', 'mentor')),
  cohort_id text not null,
  utm jsonb,
  constraint leads_email_cohort_unique unique (email, cohort_id)
);

alter table public.leads enable row level security;
-- политик для anon нет: запись только через service role из Route Handler
```

`unique (email, cohort_id)` — идемпотентность на уровне БД. Повторная отправка той
же формы вернёт ошибку `23505`, которую Route Handler превращает в честный ответ
«вы уже забронировали место», а не в 500.

`utm jsonb` — сохраняем метки из URL. На реальном проекте это первое, что спросит
маркетолог; в портфолио показывает, что вы про это думали.

## План реализации

### Шаг 1. Подготовка — 2 ч
Репозиторий, токены темы, шрифты (Cormorant + Inter + JetBrains Mono, все с
`subsets: ["cyrillic", "latin"]`), `lib/env.ts`, CI.

### Шаг 2. Модель потоков — 3 ч
- `content/cohorts.ts`, тип `Cohort`, 4 потока вперёд.
- `resolveCohortState()` — чистая функция.
- **Юнит-тесты на неё в первую очередь**, до вёрстки: набор идёт; до дедлайна 47 часов;
  осталось 2 места; набор закрыт, есть следующий; потоков не осталось; граница ровно
  в момент `enrollmentClosesAt`.

Это тот код, где ошибка не видна глазами — она проявится через месяц на проде.
Здесь тесты дешевле отладки.

### Шаг 3. Контент — 3 ч
Тексты: 8 модулей с результатами, 3 сегмента, 3 тарифа, 6 FAQ, био автора.
Подбор фото: 12 кадров для контрольного листа + герой + портрет автора.
Плёночная эстетика на Unsplash ищется по `film photography`, `darkroom`, `analog`.

### Шаг 4. Примитивы и таймер — 4 ч
- `Button`, `Input`, `Accordion` (на `radix-ui` — клавиатура и ARIA из коробки).
- `<Countdown/>` со всеми тремя решениями из раздела про ловушки.
- `/api/time` — Route Handler, отдающий `{ now: Date.now() }`, `cache: "no-store"`.
- Проверка: перевести системные часы на день вперёд — таймер не изменился.

### Шаг 5. Секции — 6 ч
Сверху вниз. Переход от тёмной части к светлой — на секции «Работы выпускников».
Контрольный лист (1.5 ч): CSS Grid, `aspect-ratio: 3/2`, подписи моноширинным,
лайтбокс переиспользуется из проекта 1.

### Шаг 6. Zustand, тарифы, липкая панель — 3 ч
- Стор с двумя полями, подписки на срезы.
- Карточки тарифов, выделение «Про», предвыбор при клике.
- Липкая панель на мобильном: `IntersectionObserver` на секции тарифов.
- Проверка в React DevTools Profiler: смена тарифа не перерисовывает таймер.

### Шаг 7. Форма и Supabase — 4 ч
- Zod-схема (email, опциональный телефон, согласие) + юнит-тесты.
- Таблица, RLS, service-role клиент в `lib/supabase/server.ts` с `import "server-only"`.
- Route Handler: валидация, honeypot, rate limit, обработка `23505` как «уже записаны».
- Сохранение UTM из `searchParams`.
- Экран успеха: что дальше, когда придёт письмо.

### Шаг 8. Полировка и деплой — 3 ч
- Blur-проявление фото, `prefers-reduced-motion`.
- Lighthouse, CLS от таймера = 0.
- JSON-LD `Course` с `hasCourseInstance` и датами потока.
- OG-картинка с датой старта, генерируемая динамически.
- Playwright: выбрать тариф «Про» → открыть форму → тариф предвыбран → отправить.
- README: отдельным разделом — как устроена модель потоков и почему таймер не верит клиенту.

**Итого: 28 ч.**

## Критерии готовности

- [ ] `resolveCohortState()` покрыт тестами на все пять состояний и граничные моменты.
- [ ] Перевод системных часов пользователя не влияет на таймер.
- [ ] Нет hydration mismatch в консоли, CLS от таймера = 0.
- [ ] Таймер не отстаёт после 10 минут в фоновой вкладке.
